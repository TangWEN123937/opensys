## 代码/执行类产出物审查清单

### 维度一：功能完成度
- [ ] 是否完成了阶段描述中要求的所有任务
- [ ] 输出文件/脚本是否实际创建且可运行
- [ ] 执行结果是否符合预期（退出码、输出内容等）
- [ ] 是否有遗漏的子任务

### 维度二：代码质量
- [ ] 代码逻辑是否正确（无明显 bug）
- [ ] 函数是否符合单一职责
- [ ] 变量命名是否清晰（望文生义）
- [ ] 是否有重复代码可提取为公共函数
- [ ] 错误处理是否完整（不要只用 bare except）
- [ ] 代码风格是否一致

### 维度三：安全检查
- [ ] 是否有硬编码的密钥/密码/令牌
- [ ] 用户输入是否做了校验和清理
- [ ] 文件操作是否检查了路径遍历风险
- [ ] 命令拼接是否有注入风险（应使用参数化）

### 维度四：架构一致性
- [ ] 新代码是否遵循项目现有模式
- [ ] 配置项是否放到了配置文件（而非硬编码）
- [ ] 是否需要更新状态定义或接口

### 维度五：输出验证
- [ ] 是否有执行成功的证据（日志、截图、输出等）
- [ ] 产出文件是否在正确的位置
- [ ] 如果修改了现有文件，是否引入了回归问题

### 评分标准
- 9-10 分：功能完整，代码质量高，安全无隐患，有验证证据
- 7-8 分：核心功能完成，有小问题但不影响使用
- 5-6 分：部分功能缺失或有明显 bug，需返工
- 3-4 分：方案可行但实现严重不足，需大量返工
- 1-2 分：方案有根本性问题，需要重新规划
