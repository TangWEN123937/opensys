## 代码审查流程（本技能被激活时执行）

当用户请求代码审查时，按以下结构化流程执行：

### 第一步：了解审查范围

1. 用 `run_terminal("find 目标目录 -name '*.py' -newer 基准文件")` 或 `run_terminal("git diff --name-only")` 确定变更文件
2. 用 `run_terminal("wc -l 变更文件")` 了解变更规模
3. 逐个用 `run_terminal("cat 文件")` 阅读完整代码

### 第二步：三维度审查

**维度一：代码质量**
- 函数是否符合单一职责（一个函数只做一件事）
- 变量命名是否清晰（能望文生义）
- 是否有重复代码可提取为公共函数
- 错误处理是否完整（不要只用 bare except）
- 注释是否与代码一致（过时注释比没有注释更糟）

**维度二：安全检查**（OpenSys 特别关注）
- 用户输入是否做了校验和清理
- 是否有硬编码的密钥/密码/令牌
- 文件操作是否检查了路径遍历风险
- 命令拼接是否有注入风险（应使用参数化）
- 敏感数据是否记入了日志

**维度三：架构一致性**
- 新代码是否遵循项目现有模式（LangGraph 节点 + 工具 + 状态）
- 配置项是否放到了 `config.py`（而非硬编码）
- 工具调用是否纳入了 `security.py` 的风险评估
- 是否需要更新 `state.py` 状态定义

### 第三步：输出审查报告

```
## 代码审查报告

**审查范围**: [文件列表和总行数]
**整体评估**: ✅ 通过 | ⚠️ 有问题需修复 | ❌ 建议重写

### 问题清单

#### 🚨 Critical（必须修复）
- [文件:行号] 问题描述 → 建议修改

#### ⚠️ Important（建议修复）
- [文件:行号] 问题描述 → 建议修改

#### 💡 Minor（可选优化）
- [文件:行号] 问题描述 → 建议修改

### 亮点
- [值得肯定的设计/实现]
```

### 审查原则

- **只提有证据的问题**：每个问题都要引用具体的文件和行号
- **给出具体建议**：不要只说"这里有问题"，要说"建议改为 XXX"
- **区分问题等级**：Critical 是功能/安全缺陷，Important 是可维护性问题，Minor 是风格建议
- **肯定好的设计**：审查不只是挑毛病，也要指出写得好的地方
