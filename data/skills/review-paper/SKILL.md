## 学术论文审查清单

### 维度一：学术规范性（权重 20%）
- [ ] 引用格式是否统一且符合目标规范（GB/T 7714 / APA / MLA）
- [ ] 参考文献列表是否完整，与正文引用一一对应
- [ ] 专业术语使用是否准确，首次出现是否有定义
- [ ] 语言风格是否符合学术语体（第三人称、客观表述、无口语化）
- [ ] 图表是否有编号、标题和来源说明
- [ ] **图表文件存在性**：正文引用的每张图/表（如「图 2-1」）是否都有对应的实际文件（如 `fig_2_1.png`）存在于 drafts/ 目录
- [ ] 摘要是否包含研究目的、方法、结果和结论四要素
- [ ] 关键词是否准确覆盖核心概念（3-5 个）

### 维度二：论证逻辑（权重 25%）
- [ ] 研究问题是否清晰明确
- [ ] 文献综述是否自然地引出研究空白和本文定位
- [ ] 研究方法是否能回答提出的研究问题
- [ ] 分析结果是否直接对应研究问题
- [ ] 结论是否由分析结果支撑（不超出证据范围）
- [ ] 各章节之间逻辑衔接是否自然
- [ ] 论证过程中是否存在逻辑跳跃或自相矛盾

### 维度三：引用完整性（权重 20%）
- [ ] 非常识性论点是否都有文献支撑
- [ ] 引用文献是否来自可信来源（学术期刊、权威专著等）
- [ ] 是否存在明显的 AI 编造内容（虚构的作者、期刊、数据等）
- [ ] 引用分布是否均匀（不集中在某一章节）
- [ ] 是否引用了该领域的经典文献和最新研究
- [ ] 直接引用和间接引用比例是否合理（直接引用不宜过多）

### 维度四：研究方法（权重 20%）
- [ ] 研究方法选择是否合理且有论证
- [ ] 数据来源和样本是否可靠、充分
- [ ] 分析过程是否可复现
- [ ] 是否讨论了方法的局限性
- [ ] 定量研究的统计方法是否恰当
- [ ] 定性研究的理论框架是否清晰

### 维度五：创新性与贡献（权重 15%）
- [ ] 研究问题是否有学术价值
- [ ] 是否有理论、方法或视角上的创新
- [ ] 对已有研究的贡献是否明确阐述
- [ ] 实践启示是否具体且有可操作性

### 评分标准
- 9-10 分：学术规范、论证严密、创新性强，可直接投稿/答辩
- 7-8 分：整体良好，个别细节需完善（引用补充、文字润色等）
- 5-6 分：有明显问题，需返工修复（逻辑漏洞、引用缺失、方法不当等）
- 3-4 分：方向正确但质量不达标，需要大范围重写
- 1-2 分：方向性错误或学术规范严重不合格，需要重新规划

### 特别关注事项

- **查重风险预判**：标记可能存在查重风险的段落（与常见表述过于相似的部分）
- **AI 痕迹检测**：检查是否存在 AI 生成的典型特征（过于流畅的废话、缺乏具体细节、引用不实等）

### 图表一致性审查（⚠️ 重点，必查）

这是写作阶段最常见的问题：正文引用了图表但没有实际生成对应文件。必须逐一核实：

- [ ] **正文→文件**：扫描所有章节中的图表引用（如「图 2-1」「表 3-1」），检查 drafts/ 目录下是否存在对应的 `fig_N_M.png` 文件。如果正文引用了「图 2-1」但 drafts/ 下没有 `fig_2_1.png`，这是**严重问题，必须扣分并在反馈中明确指出**
- [ ] **文件→正文**：drafts/ 目录下的图表文件是否都在正文中被引用（孤立文件是次要问题）
- [ ] **图表编号连续性**：同一章节内的图表编号是否连续（图 2-1、图 2-2…），不应跳号
- [ ] **Markdown 引用格式**：图表是否以 `![图 N-M 标题](文件名.png)` 格式正确插入

**审查方法**：通过产出物中的文件列表（subtasks[].output_files）和正文内容交叉比对。如果产出物中可以看到 chapter_N.md 的内容引用了「图 X-Y」，但 output_files 中没有对应的 .png 文件，则判定为图表缺失。

### 串行写作全局一致性审查（使用 executor_sequential 模式时重点检查）

- [ ] 引用编号是否全文连续递增（[1] [2] [3] ...），不应出现各章独立从 [1] 开始的情况
- [ ] 同一文献在不同章节被引用时是否使用了统一编号
- [ ] 章节编号是否顺序连贯（第一章、第二章...），无编号冲突或跳号
- [ ] 术语定义和缩写的使用是否全文一致（首次出现有定义，后续使用一致）
- [ ] 同一数据在不同章节引用时是否一致（不应出现前后矛盾的数字）
- [ ] 各章节之间是否有自然的逻辑衔接（承上启下的过渡段）

### ⛔ 审查反馈纪律（严格遵守）

- **禁止无证据质疑**：不得仅凭"感觉数据不对"就提出质疑。如果你无法指出具体哪个数据与哪个可验证来源矛盾，就不应该质疑该数据的准确性。没有实质证据的怀疑，不提
- **禁止模棱两可的反馈**：严禁使用"请自行核对""建议检查一下""可能存在问题"等含糊表述。每条反馈必须明确指出：**什么地方有什么问题，应该怎么改**
- **提了就要给方案**：凡是提出的问题，必须附带具体的解决方案或改正方向。不能只说"这里有问题"而不说怎么解决。如果你给不出解决方案，说明你也不确定是否真的有问题——那就不要提
- **质疑要有底气**：对数据、引用、结论的每一条质疑都必须基于以下至少一项实质依据：
  1. 与审查包中其他部分的数据/论述相矛盾
  2. 违反了该领域的公认常识或基本原理
  3. 存在明显的计算/逻辑错误（需指出具体错误所在）
  4. 格式/规范明确不符合审查清单中的具体条目
- **宁可漏报不可误报**：错误的质疑会导致无意义的返工，浪费执行资源。如果拿不准，宁可不提，也不要提出一个站不住脚的质疑让执行者白忙一场
- **禁止因截断判定不完整**：产出物中如有「已截断」标记，是系统因传输限制截取了显示内容，实际文件是完整的。不得因截断标记而判定「内容不完整」「缺少章节」
- **Markdown 格式限制**：产出物为 Markdown 格式，不支持页码、页眉页脚、分页符、目录页码等排版功能，这些是后续排版阶段的工作，写作阶段不得因缺少这些而扣分

### 审查报告文件

审查结果会自动保存为 `<任务目录>/drafts/review_report.md`，每次审查覆盖更新，始终只保留最新一份。报告包含：
- 审查时间、阶段、结论、评分
- 总评、问题列表、改进建议
- 子任务审查详情（表格形式）
