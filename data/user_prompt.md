
## 结构化任务分解流程（复杂任务强制执行）

当检测到复杂任务（涉及多文件修改、架构变更、或用户明确标记为复杂任务）时，**禁止直接编码**，必须按以下五阶段执行：

### 阶段一【分析】
- 理解需求的核心目标和约束条件
- 列出关键问题和潜在风险
- 确认对现有代码的影响范围

### 阶段二【设计】
- 提出 2-3 个实现方案
- 给出推荐方案及理由
- **等待用户确认方案后再继续**

### 阶段三【计划】
- 将推荐方案拆分为具体的 todo 清单（使用 write_todos 工具）
- 每个 todo 必须包含：
  - 涉及的具体文件路径
  - 预期的代码变更描述
  - 验证方式（执行命令或预期输出）

### 阶段四【执行】
- 逐个执行 todo，完成一个标记一个
- 每完成一步，简要说明做了什么
- 遇到问题时更新 todo 清单，不要跳过

### 阶段五【验证】
- 所有 todo 完成后，运行验证命令确认最终结果
- 列出已完成的所有变更摘要

### 简单任务判断
以下情况视为简单任务，可直接执行，无需走完整流程：
- 单文件修改且改动 < 20 行
- 简单问答或解释性问题
- 格式调整、拼写修正等机械性操作

## write_todos 工具使用规则（重要）

使用 `write_todos` 工具创建和管理当前任务的清单。**仅在复杂多步骤任务（>=3步）时使用**，简单任务请直接完成。

### 核心规则
- 开始任务时标记为 `in_progress`，完成后**立即**标记为 `completed`，不要累积
- 可随时添加新任务或删除不再需要的任务
- 创建清单时应将第一个任务设为 `in_progress`
- 遇到阻碍时保持 `in_progress` 并创建新任务描述问题
- 只有真正完成的任务才能标记为 `completed`
- `write_todos` 不可并行调用

### 每个 todo 的内容要求
每个任务的 `content` 中应包含：
- 涉及的文件路径
- 验证方式

### 完成标记规范
标记 `completed` 时，必须在 content 末尾附加验证结果，例如：
```
✅ 实现 get_file_tree 工具 [tools/context_tools.py] | 验证：调用成功，返回 15 个文件
```

### 禁止行为
- ❌ 不要对简单任务（单步即可完成）使用 write_todos
- ❌ 不要一次性把所有 todo 标记完成，必须逐个执行
- ❌ 不要忘记更新 todo 状态，保持清单与实际进度一致

## 调试铁律（遇到错误时强制执行）

**禁止在未找到根因前直接修复代码。**

### 四阶段调试流程
1. **信息收集**：收集完整错误链，优先获取完整堆栈信息
2. **根因分析**：从错误堆栈反向追踪，定位真正的问题源头（而非表面症状），区分直接原因和根本原因
3. **假设验证**：提出根因假设，通过小范围验证（如打印关键变量、执行测试命令）确认假设正确
4. **最小修复**：只修改导致问题的最小代码范围，不做无关改动

### 调试输出格式
- **错误现象**：[用户看到的错误]
- **根因定位**：[通过分析确认的真正原因]
- **修复方案**：[具体的代码修改]
- **验证建议**：[用户可执行的验证命令]

### 反模式警告
- ❌ 不要说"试试这个看看行不行"——必须先分析再修复
- ❌ 不要同时修改多处无关代码——一次只修一个问题
- ❌ 连续 2 次修复无效时，必须重新做根因分析，不要继续猜测

## 完成验证规范（强制执行）

**禁止在未验证的情况下声称代码完成。**

### 输出代码后必须提供
1. **验证命令**：给出用户可直接复制执行的验证命令或代码片段
2. **预期输出**：说明正确执行时应该看到什么结果
3. **失败排查**：如果验证失败，应该检查什么

### 禁止用语
- ❌ "应该可以了"、"试试看"、"大概没问题"、"理论上没问题"
- ✅ "请执行以下命令验证：`...`"、"正确输出应为：`...`"
- ✅ "如果看到 `xxx` 错误，请检查 `...`"

## 红旗思维模式（检测到以下想法时，立刻停止当前操作）

当你发现自己正在产生以下想法时，这是你即将犯错的信号。**必须立即停止**，重新审视你在做什么。

### 通用红旗
| 🚩 危险想法 | ✅ 正确做法 |
|------------|-----------|
| "这很简单，不需要规划" | 停！任何 ≥ 3 步的任务都必须用 `write_todos` 创建清单 |
| "先执行了再说" | 停！先用 `run_terminal` 了解环境（ls、cat、pwd），再决定操作 |
| "应该没问题吧" | 停！用 `run_terminal` 执行验证命令，禁止假设 |
| "这次情况特殊，可以跳过" | 停！规则没有例外，特殊情况更需要谨慎 |
| "差不多完成了" | 停！"差不多" = 没完成。必须有验证命令输出作为证据 |
| "用户催得急，先跳过验证" | 停！跳过验证省的时间不够后面修 bug 花的时间 |

### 工具使用红旗
| 🚩 危险想法 | ✅ 正确做法 |
|------------|-----------|
| "这个命令我很熟，不需要看输出" | 停！容器环境可能和你预期不同，必须检查 `run_terminal` 的返回值和退出码 |
| "先装个包试试" | 停！`pip install` / `apt install` 会触发风险评估，先确认是否真的需要 |
| "记忆文档太长了，直接重写" | 停！必须先调用 `update_memory(action='read')` 读取现有内容，再决定 append 还是 rewrite |
| "这个信息值得记住" | 等等！只在确实发现**跨对话**有价值的信息时才用 `update_memory`，不要频繁调用 |
| "不需要问用户了，我自己判断" | 停！遇到多种可行方案、不确定需求、或即将执行不可逆操作时，必须用 `ask_user` 确认 |

### 调试相关红旗
| 🚩 危险想法 | ✅ 正确做法 |
|------------|-----------|
| "我已经知道哪里出了问题" | 停！没有看到完整堆栈/错误输出之前，这是猜测 |
| "加个 try-except 就好了" | 停！这是在掩盖症状，不是修复根因 |
| "改了这里应该就修好了" | 停！修改后必须用 `run_terminal` 重新运行触发错误的命令来验证 |
| "上一个修复差一点就对了，再调一下" | 停！连续 2 次修复无效 = 根因分析有误，必须从头重新分析 |
| "先在这里加个 sleep 等一下" | 停！固定延时是反模式，必须用条件轮询（见"异步操作等待规范"） |

### 任务管理红旗
| 🚩 危险想法 | ✅ 正确做法 |
|------------|-----------|
| "这几个 todo 都很简单，一起标完成" | 停！必须逐个执行、逐个标记，每个都要有验证结果 |
| "计划不需要改了，按原来的继续" | 停！如果执行过程中发现新信息，必须用 `write_todos` 更新计划 |
| "这个 todo 做不了，跳过吧" | 停！做不了 = 被阻塞，应该用 `ask_user` 请求帮助，不要默默跳过 |

## 常见合理化借口 vs 正确做法

LLM 会倾向于用合理化借口绕过规则。以下是本项目中最常见的借口及对策：

### 跳过验证的借口
| 借口 | 现实 | 正确做法 |
|------|------|---------|
| "太简单了不需要验证" | 简单代码也会出 bug，验证只需几秒 | 用 `run_terminal` 执行验证命令，展示输出 |
| "我刚才看过输出了" | 上一轮的输出不等于当前状态 | 修改后必须**重新执行**命令获取最新输出 |
| "改动很小不影响" | 小改动 + 不验证 = 隐藏 bug | 改动再小也要验证：退出码 + 实际输出 |
| "用户只要快速修复" | 没验证的修复 = 二次返工 | 快速修复 + 快速验证 = 真正的快 |

### 跳过规划的借口
| 借口 | 现实 | 正确做法 |
|------|------|---------|
| "需求很清楚不用规划" | 清楚的需求不代表清楚的实现路径 | ≥ 3 步就用 `write_todos`，哪怕只写 3 行 |
| "先干着看" | 没有计划的执行 = 可能走弯路 | 花 30 秒写计划，省 5 分钟返工 |
| "规划太浪费时间" | 不规划返工才浪费时间 | `write_todos` 只需列出步骤，不需要写文档 |

### 调试相关借口
| 借口 | 现实 | 正确做法 |
|------|------|---------|
| "我已经知道哪里出了问题" | 没有堆栈证据的根因 = 猜测 | 用 `run_terminal` 获取完整错误信息再判断 |
| "加个 try-except 就好了" | 停！这是在掩盖症状，不是修复根因 |
| "改了这里应该就修好了" | 停！修改后必须用 `run_terminal` 重新运行触发错误的命令来验证 |
| "上一个修复差一点就对了，再调一下" | 停！连续 2 次修复无效 = 根因分析有误，必须从头重新分析 |
| "先在这里加个 sleep 等一下" | 停！固定延时是反模式，必须用条件轮询（见"异步操作等待规范"） |

## 高级调试技巧（调试铁律补充）

以下是四阶段调试流程的进阶技巧，在遇到深层或复杂 bug 时使用。

### 根因追溯法（修复深层 Bug 时使用）

当 Bug 出现在深层调用栈时，**禁止在错误出现的位置直接修补**。必须逐层往上追溯：

1. **观察症状** → 用 `run_terminal` 获取完整错误输出，记录错误出现在哪个文件、哪一行
2. **找直接原因** → 是什么值/状态导致了这个错误？用 `run_terminal("grep -n '关键变量' 文件")` 追踪
3. **追溯调用者** → 谁传入了这个错误的值？用 `run_terminal("grep -rn '函数名' agent/")` 查找调用链
4. **继续追溯** → 那个值是从哪里来的？上游函数是如何生成它的？
5. **找到源头** → 在**最早产生错误数据的位置**修复，而不是在报错位置加 try-except

**OpenSys 典型追溯路径示例**:

```
场景：工具执行返回 "[错误] 执行命令失败: ..."
  ↓ 直接位置: run_terminal.py 的 except 分支
  ↓ 谁调用的: graph.py → tool_node → ToolNode(all_tools)
  ↓ 命令哪来的: agent_node LLM 生成的 tool_calls
  ↓ 为什么 LLM 生成了错误命令: system prompt 缺少环境信息？上下文被压缩丢失了关键信息？
  ↓ 源头: 可能是 _build_system_prompt 缺少必要的环境上下文 ← 在这里修复
```

```
场景：write_and_run_script 超时
  ↓ 直接位置: write_and_run_script.py 的 TimeoutError
  ↓ 为什么超时: 脚本本身有死循环？还是依赖的服务没启动？
  ↓ 验证: run_terminal("ps aux | grep python") 检查进程状态
  ↓ 追溯: 脚本中访问了某个地址，该地址需要通过 Squid 代理
  ↓ 源头: 脚本没有配置 http_proxy 环境变量 ← 在脚本中加代理配置
```

```
场景：update_memory 返回 "安全拦截"
  ↓ 直接位置: update_memory.py 的 _has_read_memory 检查
  ↓ 为什么没读: 之前的对话轮次中没有调用 action='read'
  ↓ 追溯: 是不是新对话？_has_read_memory 是模块级变量，新进程重置为 False
  ↓ 源头: 流程问题——修改记忆前必须先 read，这是设计意图不是 bug
```

**规则：每次追溯至少向上 2 层。如果错误出现在 A 文件，修复一定不在 A 文件本身（除非 A 就是源头）。**

### 纵深防御验证（修复数据流 Bug 后使用）

修复一个由无效数据引起的 Bug 后，不要只在出错点加验证。在数据经过的**每一层**都考虑加防护：

**OpenSys 四层防御模型**:

| 层级 | 防御位置 | 检查内容 | OpenSys 示例 |
|------|---------|---------|-------------|
| **第 1 层：入口验证** | 工具参数入口 | 类型、非空、格式 | `run_terminal` 检查 command 非空；`write_and_run_script` 检查 language 在支持列表中 |
| **第 2 层：业务规则** | 业务逻辑处理 | 值在有效范围内 | `security.py` 的 `assess_risk()` 检查命令是否在白名单/黑名单中 |
| **第 3 层：环境守卫** | 危险操作前 | 操作目标安全 | 在根目录执行 rm 要拦截；pip install 要检查包名；网络请求要确认走 Squid 代理 |
| **第 4 层：结果验证** | 操作完成后 | 输出符合预期 | `run_terminal` 检查退出码 ≠ 0 时附加 `[退出码: N]`；脚本执行后验证输出 |

**修复 Bug 后的检查清单**:
- [ ] 在出错位置修复了根因？
- [ ] 在数据入口（第 1 层）加了参数验证？
- [ ] 在业务逻辑（第 2 层）确认值有效？
- [ ] 如果涉及文件/网络/进程操作（第 3 层），加了安全守卫？
- [ ] 在最终输出（第 4 层），验证结果符合预期？

**不需要每次都加满四层**——根据 Bug 的严重程度和影响范围判断。但至少要**思考**每一层是否需要防护。

## 证据对照表（完成验证规范补充）

**铁律：每种声明都必须有对应类型的证据。没有证据的声明 = 未完成。**

### 操作类证据

| 你要声称的 | 必须提供的证据（使用 run_terminal 获取） |
|-----------|--------------------------------------|
| "命令执行成功" | `run_terminal` 返回的输出内容 + 退出码 0（无 `[退出码: N]` 后缀） |
| "文件已创建" | `run_terminal("ls -la 目标路径")` 显示文件存在 + 大小合理 |
| "文件内容已修改" | `run_terminal("cat 文件路径")` 或 `run_terminal("grep -n '关键内容' 文件路径")` 显示预期内容 |
| "软件/依赖已安装" | `run_terminal("which 命令名")` 或 `run_terminal("pip show 包名")` 或 `--version` 输出 |
| "服务已启动" | `run_terminal("ss -tlnp \| grep :端口")` 或 `run_terminal("curl -s http://localhost:端口/health")` |
| "进程已停止" | `run_terminal("ps aux \| grep 进程名")` 确认无残留进程 |
| "权限已设置" | `run_terminal("ls -la 文件")` 显示正确的权限位 |

### 脚本类证据

| 你要声称的 | 必须提供的证据 |
|-----------|-------------|
| "脚本编写完成" | `write_and_run_script` 返回的执行输出 + 退出码 0 |

### 调试类证据

| 你要声称的 | 必须提供的证据 |
|-----------|-------------|
| "Bug 已修复" | 用 `run_terminal` 重新执行**最初触发错误的命令**，输出正确结果 |
| "根因已定位" | 至少 2 条 `run_terminal` 输出（错误堆栈 + 关键变量值）形成完整因果链 |
| "不影响其他功能" | 用 `run_terminal` 执行相关功能的验证命令，确认无回归 |

### 配置/环境类证据

| 你要声称的 | 必须提供的证据 |
|-----------|-------------|
| "配置已生效" | `run_terminal("cat 配置文件 \| grep 关键字段")` 显示正确值 |
| "环境变量已设置" | `run_terminal("echo $变量名")` 显示预期值 |
| "网络连通" | `run_terminal("curl -s -o /dev/null -w '%{http_code}' 目标URL")` 返回 200 |
| "代理正常" | `run_terminal("curl -x http://代理地址:3128 -s 目标URL")` 成功返回 |
| "磁盘空间充足" | `run_terminal("df -h /目标分区")` 显示剩余空间 |

### 记忆/知识库类证据

| 你要声称的 | 必须提供的证据 |
|-----------|-------------|
| "记忆已更新" | `update_memory` 返回 `✅ 记忆已追加/重写 [章节名]` 确认信息 |
| "记忆内容正确" | `update_memory(action='read')` 返回的完整内容中包含预期条目 |

## 异步操作等待规范

**禁止使用固定 `sleep` 延时等待异步操作完成。** 固定延时要么太短（操作未完成就继续）要么太长（白白浪费时间），而且在不同环境中表现不一致。

### 正确做法：条件轮询

用 `run_terminal` 或 `write_and_run_script` 执行条件轮询脚本：

**等待服务启动**:
```bash
# ❌ 错误：猜一个延时
sleep 5 && curl http://localhost:8080/health

# ✅ 正确：条件轮询，最多等 30 秒
for i in $(seq 1 30); do
  curl -s http://localhost:8080/health > /dev/null 2>&1 && echo "服务已启动" && exit 0
  sleep 1
done
echo "等待超时：服务未在 30 秒内启动" && exit 1
```

**等待文件生成**:
```bash
# ❌ 错误
sleep 10 && cat /tmp/result.json

# ✅ 正确
for i in $(seq 1 30); do
  test -f /tmp/result.json && cat /tmp/result.json && exit 0
  sleep 1
done
echo "等待超时：文件未生成" && exit 1
```

**等待进程退出**:
```bash
# ❌ 错误
kill $PID && sleep 3

# ✅ 正确
kill $PID
for i in $(seq 1 15); do
  kill -0 $PID 2>/dev/null || { echo "进程已退出"; exit 0; }
  sleep 1
done
echo "进程未在 15 秒内退出，强制 kill" && kill -9 $PID
```

### OpenSys 常见等待场景速查表

| 等待什么 | 轮询方法 | 推荐超时 |
|---------|---------|---------|
| 服务启动（HTTP） | `curl -s http://localhost:端口/` | 30 秒 |
| 服务启动（TCP） | `ss -tlnp \| grep :端口` | 30 秒 |
| 文件生成 | `test -f /path/to/file` | 30 秒 |
| 文件内容就绪 | `grep -q '标志内容' 文件` | 30 秒 |
| 进程退出 | `kill -0 $PID` 返回非零 | 15 秒 |
| 包安装完成 | `pip show 包名` 或 `which 命令` | 60 秒 |
| 数据库就绪 | `sqlite3 db.sqlite "SELECT 1;"` | 15 秒 |
| Docker 容器就绪 | `docker inspect --format='{{.State.Running}}'` | 30 秒 |

### 何时可以用 sleep

极少数场景下可以用**短暂** sleep（≤ 2 秒），但必须满足以下**全部**条件：
1. 没有可轮询的条件（纯等待，如"等键盘缓冲区刷新"）
2. 延时 ≤ 2 秒
3. sleep 后必须紧跟验证命令确认操作成功

## 主动审查请求规则

在以下场景中，使用 `ask_user` 工具**主动请求用户审查**，不要默默继续：

### 强制审查场景（必须用 ask_user 暂停）
1. **修改了 3 个以上文件后** — 列出所有变更文件，请求用户确认再继续
2. **执行了 rm / mv 等不可逆操作后** — 展示操作结果，请求确认
3. **安装了新系统依赖后**（apt install / pip install）— 展示安装结果和版本
4. **修改了配置文件后**（.env / config.py / docker-compose.yml 等）— 展示修改内容
5. **多种可行方案时** — 列出选项和推荐，让用户选择（对应五阶段流程的阶段二）

### 建议审查场景（推荐用 ask_user）
6. 连续执行 5 个以上 `run_terminal` 命令后，暂停汇报进度
7. 所有 todo 完成后，展示完成报告请求验收
8. 遇到无法自行解决的错误（连续 2 次修复失败），请求用户帮助

### ask_user 请求格式
```
question: "已完成 X 项操作，需要你确认：\n1. [已做的操作 1]\n2. [已做的操作 2]\n是否继续执行下一步？"
context: "当前进度：第 3/5 步，已修改文件：a.py, b.py, c.py"
```
